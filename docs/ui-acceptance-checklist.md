# FlowCast Demo UI Acceptance Checklist

使用方式：

- `P0`：内部 Demo 演示必须通过，失败即阻断验收。
- `P1`：建议通过，失败不阻断首轮视图评审，但需要记录。
- `目标态`：后续接真实能力或细化页面时补齐，不作为当前 Android 本地 Mock 的硬性阻断。

当前依据：仓库未提供 `docs/final-ui-design.md`，本清单以 `docs/final-prd.md`、`docs/screen-map.md`、`docs/api-contracts.md`、Android Compose 本地 Mock 和 backend FastAPI Mock 为准。

## 1. Global

- [ ] P0 App 名称显示为 `闪阅 FlowCast`，或在关键页面有明确品牌/模块露出。
- [ ] P0 底部导航包含 `视频 / 新闻 / 收银台 / 我的` 四个入口。
- [ ] P0 App 默认进入或可立即到达 `视频` Tab。
- [ ] P0 断开 FastAPI 后，视频、新闻、支付、NFC 仍能展示本地 Mock 内容。
- [ ] P0 页面文案以中文为主，必要技术字段、接口名和 Scheme 可保留英文。
- [ ] P0 所有核心卡片、按钮和返回入口可点击且不崩溃。
- [ ] P1 页面主色使用深墨绿/青绿，金额和高价值信息使用暖金或其他明确强调色。
- [ ] P1 卡片、按钮、信息块圆角和间距一致，页面看起来像产品 Demo 而不是调试页。
- [ ] P1 二级页面返回后，用户能继续完成原模块或切换其他 Tab。

## 2. 视频 Feed

页面目标：让用户第一眼感到这是短视频沉浸浏览页。

- [ ] P0 可通过底部 `视频` Tab 进入。
- [ ] P0 页面使用深色或沉浸式背景，内容区域接近竖屏视频卡。
- [ ] P0 每条视频至少展示标题、作者、来源标识。
- [ ] P0 展示点赞、评论、分享等动作入口或明确占位。
- [ ] P0 展示数据源状态，例如 `Mock`、`公开视频源`、`Public Video API placeholder` 或 `FastAPI`。
- [ ] P0 视频不可播放时，有封面、渐变、播放图标或可识别的播放占位。
- [ ] P0 页面不应退化成普通文本列表。
- [ ] P1 顶部频道或标签能体现内容消费场景，例如推荐、AI、零售、NFC、支付。
- [ ] P1 后续接 Media3 时，当前标题/作者/互动/来源的信息层级不需要重做。

## 3. 新闻列表

页面目标：快速扫读 AI 新闻，明确标题、概要和来源。

- [ ] P0 可通过底部 `新闻` Tab 进入。
- [ ] P0 顶部明确当前频道为 AI 新闻或科技资讯。
- [ ] P0 每张新闻卡片展示标题、概要、来源、发布时间。
- [ ] P0 标题比概要更突出，概要不堆长段落。
- [ ] P0 卡片点击后进入新闻详情页。
- [ ] P0 列表为空或接口失败时有 Mock 兜底或空态提示。
- [ ] P0 页面整体为清爽浅色阅读风格，不使用视频页的沉浸黑背景。
- [ ] P1 频道筛选或标签存在时，选中态和未选中态可区分。
- [ ] P1 新闻列表至少有 2 条内容，便于评审者理解列表形态。

## 4. 新闻详情

页面目标：展示新闻内容总结，并在结尾给出原文链接。

- [ ] P0 顶部展示新闻标题和来源/发布时间信息。
- [ ] P0 主体展示 AI 总结，不展示未授权全文转载。
- [ ] P0 至少展示 3 条关键点，便于评审者快速理解内容。
- [ ] P0 页面展示原文链接、链接信息块或打开原文按钮。
- [ ] P0 详情页提供返回入口。
- [ ] P1 原文链接点击后有明确跳转行为或模拟跳转反馈。
- [ ] P1 返回新闻列表后，底部导航状态仍保持在新闻模块或能清楚回到新闻上下文。

## 5. 收银台

页面目标：模拟真实收银台，金额优先，支付方式清晰。

- [ ] P0 可通过底部 `收银台` Tab 进入。
- [ ] P0 可由 NFC 收款 Scheme 解析页进入或跳回收银台。
- [ ] P0 页面突出展示订单金额。
- [ ] P0 展示商户名称、订单号或业务场景。
- [ ] P0 至少提供模拟支付宝、模拟微信、模拟银联三种方式。
- [ ] P0 页面展示当前收银台 Scheme 或订单来源，方便 Demo 解释。
- [ ] P0 当前支付方式有选中态。
- [ ] P0 主按钮文案明确，例如 `确认支付` 或 `使用模拟微信支付`。
- [ ] P0 点击主支付按钮进入对应 MockPaymentScreen。
- [ ] P1 金额使用高强调样式，明显高于其他订单字段。

## 6. 模拟支付

页面目标：模拟第三方支付界面和回跳行为。

- [ ] P0 页面标题或品牌区明确当前模拟渠道：支付宝、微信或银联。
- [ ] P0 展示订单金额。
- [ ] P0 展示支付 Scheme、订单号或其他可解释当前交易的信息。
- [ ] P0 提供 `支付成功`、`支付失败`、`取消支付` 三个结果动作。
- [ ] P0 三个动作都能跳转到 PaymentResultScreen。
- [ ] P0 返回或取消时不应造成导航栈卡死。
- [ ] P1 页面视觉应与收银台区分，但不冒充真实支付 App。
- [ ] P1 可在文案中说明“仅模拟支付结果，不接真实渠道”。

## 7. 支付结果

页面目标：明确告诉用户支付最终状态和下一步。

- [ ] P0 支持成功、失败、取消状态展示。
- [ ] P0 成功状态使用积极颜色和明确文案。
- [ ] P0 失败/取消状态与成功状态可区分。
- [ ] P0 提供返回收银台或继续操作入口。
- [ ] P0 从结果页返回后，用户能继续浏览其他 Tab。
- [ ] P1 展示订单号、金额、支付方式中的至少两个关键信息。
- [ ] P1 支持处理中状态的展示口径。
- [ ] 目标态 处理中状态能通过 `/payments/status/{order_id}` 回查或刷新。

## 8. NFC Demo

页面目标：把 NFC 设计成场景化业务入口。

- [ ] P0 可从 `我的` 页面进入。
- [ ] P0 页面包含三类场景：线下商户收款、门店桌牌点餐、小程序跳转。
- [ ] P0 每个场景展示简短业务说明和对应 Scheme。
- [ ] P0 每个场景可点击触发解析或跳转。
- [ ] P0 页面应突出“碰一碰触发”或 NDEF URI 读取后的 Demo 语境。
- [ ] P0 不应只展示裸字符串列表。
- [ ] P1 每个场景有可区分标签，例如收款、点餐、小程序。

## 9. NFC 解析

页面目标：解释 NFC Scheme 如何映射到业务动作。

- [ ] P0 展示原始 Scheme。
- [ ] P0 展示业务动作说明：进入收银台、进入点餐页/占位说明、进入小程序跳转页/占位说明。
- [ ] P0 线下收款场景可从解析页进入收银台。
- [ ] P0 桌牌点餐和小程序跳转场景至少展示下一步占位说明，不要求真实页面。
- [ ] P0 从解析页可以返回 NFC Demo 或继续完成后续动作。
- [ ] P1 展示解析后的关键字段，例如 merchantId、amount、storeId、tableId、platform、appId、path。
- [ ] 目标态 无法识别的 Scheme 应展示错误提示，不应崩溃。

## 10. 我的

页面目标：承载 Demo 信息和工具入口。

- [ ] P0 展示 App 名称、内部 Demo 标识或当前版本说明。
- [ ] P0 展示能力概览：短视频、AI 新闻、模拟支付、NFC。
- [ ] P0 提供 NFC Demo 入口。
- [ ] P1 可提供订单记录、NFC 记录等未来入口，但第一版允许为占位。
- [ ] P1 页面风格与新闻/支付保持一致，避免变成设置项堆叠。

## 11. Backend Contract Smoke

页面验收不依赖后端在线，但 FastAPI Mock 若启动，应满足以下契约：

- [ ] P0 `GET /health` 返回 `status=ok`，并可附带 service、version、mode、timestamp 等 Mock 元信息。
- [ ] P0 `GET /videos/feed` 返回 `items` 和 `next_cursor`，每项包含 title、author、source、cover_url/play_url 或可解释占位。
- [ ] P0 `GET /news` 返回新闻列表；`GET /news/{id}` 返回 summary、key_points、source_url。
- [ ] P0 `POST /orders` 返回 pending 订单、amount、merchant 信息和 cashier_scheme。
- [ ] P0 `GET /payments/methods` 返回 alipay、wechat、unionpay。
- [ ] P0 `POST /payments/mock` 能把订单更新为 success、failed 或 canceled。
- [ ] P0 `GET /payments/status/{order_id}` 返回 order_id、status、method、amount。
- [ ] P0 `GET /nfc/scenarios` 返回 payment、order、miniapp 三类场景和 Scheme。

## 12. Demo Walkthrough

评审演示必须能按以下路径完成：

```text
视频 Tab
  -> 新闻 Tab
  -> 新闻详情
  -> 返回新闻
  -> 收银台
  -> 选择一种模拟支付方式
  -> 模拟支付成功/失败/取消任一状态
  -> 支付结果
  -> 我的
  -> NFC Demo
  -> 线下收款 Scheme
  -> Scheme 解析
  -> 收银台
```

通过标准：

- [ ] P0 全流程无崩溃。
- [ ] P0 每一步页面目标清晰。
- [ ] P0 所有 Mock 状态都能被解释为第一版 Demo 行为。
- [ ] P0 评审者不看代码也能说出四条主路径：视频、新闻、支付、NFC。
- [ ] P1 再走一遍 NFC 桌牌点餐和小程序场景，确认占位动作说明清楚。
